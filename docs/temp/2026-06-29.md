# Changes — 2026-06-29

## TFLite Hero Classifier Integration (TD-15)

Integrated the trained model `mlbb_hero_classifier.tflite` into the MLBB Draft Assistant
Android codebase as the **primary** hero-portrait matching path in `PortraitMatcher`,
replacing the pHash-only approach (pHash + histogram becomes the fallback).

---

## New files

### `app/src/main/java/com/mlbb/assistant/capture/HeroClassifier.kt`
Full TFLite `Interpreter` wrapper around `mlbb_hero_classifier.tflite`.

- Memory-maps the model from assets via `MappedByteBuffer` + `FileInputStream`/`FileChannel`
  (required for NNAPI/GPU delegate safety and zero-copy model loading).
- Preprocessing: `Bitmap → [1, 224, 224, 3]` float32 tensor, pixels normalised to `[-1, 1]`
  (`pixel / 127.5 − 1.0`) — standard MobileNetV3 input convention.
- Inference: single `Interpreter.run()` call → `[1, 120]` softmax float array.
- Returns top-K `ClassificationResult(heroId, confidence)` pairs sorted by descending confidence.
- `isAvailable` guard: returns `false` (forces pHash fallback) if the model fails to load,
  the labels file is missing, or the label count does not equal the model output size.
- Implements `Closeable`; `close()` releases the native `Interpreter`.

### `app/src/main/assets/hero_classifier_labels.txt`
120-line label file mapping model output index (line number, 0-based) → heroId.

- IDs 1–120 in ascending order, matching the class ordering used during training.
- Comment header (`#`) explains the format and the retraining contract:
  if the model is retrained with a different class ordering, update this file to match.
- Blank lines and `#` comment lines are skipped by `HeroClassifier` at load time.

## Modified files

### `app/src/main/java/com/mlbb/assistant/capture/PortraitMatcher.kt`
Rewritten to add TFLite as the primary matching path.

**New field:**
```kotlin
private val classifier = HeroClassifier(context)
```

**`match()` logic (new dual-path flow):**
1. **TFLite primary** — calls `classifier.classify(slotBitmap, topK = TFLITE_TOP_K)`.
   - Top-1 confidence ≥ `TFLITE_ACCEPT_THRESHOLD` (0.70): return result (respecting
     multi-frame `consecutiveHits` counter).
   - Top-1 confidence ∈ `[TFLITE_TENTATIVE_THRESHOLD, TFLITE_ACCEPT_THRESHOLD)` (0.45–0.70):
     return with `requiresConfirmation = true`.
   - Top-1 heroId not in `availableHeroes` (already picked/banned) or confidence < 0.45:
     fall through to pHash.
2. **pHash + histogram fallback** — dHash pre-filter (similarity < 0.60 → skip), then
   `pHash × 0.65 + 48-bin Bhattacharyya histogram × 0.35` hybrid score.

**Other changes:**
- `close()` now delegates to `classifier.close()`.
- `preloadHashes()` logs when classifier is available (pHash preload kept as fallback insurance).
- `applyConfirmation()` extracted as a private helper shared by both paths.
- Histogram bins increased to 16 per channel (48-element vector, Bhattacharyya similarity).

### `app/src/main/java/com/mlbb/assistant/capture/PhaseDetectionConfig.kt`
Added three TFLite threshold constants:

```kotlin
const val TFLITE_ACCEPT_THRESHOLD: Float    = 0.70f
const val TFLITE_TENTATIVE_THRESHOLD: Float = 0.45f
const val TFLITE_TOP_K: Int                 = 3
```

### `app/src/main/java/com/mlbb/assistant/capture/HeroPortraitObjectDetector.kt`
Rewritten stub with a clarifying KDoc table distinguishing the two ML tasks:

| Task | Class | Status |
|---|---|---|
| **Classification** — which hero in a pre-cropped slot | `HeroClassifier` | ✅ Integrated |
| **Detection** — where hero portrait regions are in the raw frame | `HeroPortraitObjectDetector` | ⚙️ Stubbed |

### `gradle/libs.versions.toml`
- Added version pin: `tensorflowLite = "2.16.1"`
- Added library alias:
  ```toml
  tensorflow-lite = { group = "org.tensorflow", name = "tensorflow-lite", version.ref = "tensorflowLite" }
  ```

### `app/build.gradle.kts`
- Added runtime dependency:
  ```kotlin
  implementation(libs.tensorflow.lite)
  ```
- Added `androidResources` block inside `android {}`:
  ```kotlin
  androidResources {
      noCompress += listOf("tflite")
  }
  ```
  **Why this matters:** TFLite's `Interpreter` memory-maps the model file directly from the APK
  via `FileChannel.map()`. If aapt compresses the asset (default for unknown extensions),
  the map call throws `IOException: offset + length > FileChannel size`. The `noCompress` rule
  stores the `.tflite` file uncompressed in the APK.

## Model specification (reference)

| Property | Value |
|---|---|
| Architecture | MobileNetV3Small (Keras), converted via TF 2.20.0 |
| Input shape | `[1, 224, 224, 3]` float32 |
| Input normalisation | `pixel / 127.5 − 1.0` → range `[-1, 1]` |
| Output shape | `[1, 120]` float32 softmax |
| Output mapping | Line *n* of `hero_classifier_labels.txt` = heroId for output neuron *n* |
| Hero coverage | IDs 1–120 (heroes 121–132 route to pHash fallback) |
| Inference threads | 2 (`Interpreter.Options.setNumThreads(2)`) |
| Asset size | ~2 MB |

## Known limitations / future work

- Heroes added after training (IDs 121–132 in the current roster) have no TFLite coverage
  and are silently matched via the pHash fallback.
- The SSD/YOLO portrait *region* detector (`HeroPortraitObjectDetector`) remains stubbed.
  Training requires 500+ annotated portrait crops (see `roadmap.md` RA-05 backlog).
- `Bitmap.createScaledBitmap` uses bilinear interpolation; bicubic would be more accurate
  but was not used to avoid additional dependencies.
- `Interpreter` is not thread-safe — callers must not invoke `match()` concurrently on the
  same `PortraitMatcher` instance without external synchronisation.
