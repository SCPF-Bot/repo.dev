## 🔄 The 4-Pass Review & Fix Cycle (Mandatory)

Every time you are asked to review a file, a PR, or the whole module, you must execute these 4 distinct passes in **strict order**. Do not skip any pass.

### Pass 1: Structural Ordering (Alphabetical & Logical)
- Reorder `import` statements alphabetically (ascending A→Z). Merge duplicate imports.
- Within each class/file, reorder declarations in a predictable, natural order:
  1. Companion object / constants (`const val`)
  2. Injected dependencies (`@Inject` / constructor params)
  3. Public properties (`val`/`var`)
  4. Private properties
  5. Public functions (alphabetically, or grouped by visibility)
  6. Private functions (alphabetically)
- Ensure package names follow a logical alphabetical depth (e.g., `com.example.feature.login` before `com.example.feature.signup` if they are siblings).
- For data classes, order primary constructor parameters alphabetically where it makes sense (to avoid merge conflicts).

### Pass 2: Gradle Initialisation, Configuration & Execution
- Examine `settings.gradle.kts` – verify all modules are correctly included (`include(":app", ":core", ":feature:login")`).
- Examine root `build.gradle.kts` – check plugin management (`pluginManagement`), repositories, and global buildscript classpaths.
- Examine each module’s `build.gradle.kts`:
  - Ensure all plugin IDs are correctly applied.
  - Verify that dependencies are declared via `libs.versions.toml` (never hardcoded).
  - Check that tasks are configured lazily (using `configure` or `named` to avoid configuration overhead).
- Review `gradle.properties` – ensure JVM args, parallel execution, and configuration caching are set appropriately for performance.
- Validate the Gradle wrapper (`gradle/wrapper/gradle-wrapper.properties`) matches the required distribution version for the project.

### Pass 3: App Lifecycle (Launch → Close)
- Trace the critical path from Application start (`attach()`, `onCreate()`) through the main Activity/Fragment lifecycles to the first rendered frame.
- Verify correct `super` calls in every lifecycle method (e.g., `onCreate()`, `onStart()`, `onResume()`, `onPause()`, `onStop()`, `onDestroy()`).
- Check that `ViewModel`s are scoped correctly (`viewModelScope`, `activityViewModels()`) and survive configuration changes.
- Validate Process Death handling – ensure `SavedStateHandle` or `onSaveInstanceState()` is used to retain critical UI state across kills.
- Ensure proper resource cleanup in `onDestroy()` / `onCleared()` (e.g., unregister listeners, cancel coroutine jobs, close database connections).
- Flag any potential memory leaks (e.g., registering observers without clearing them).

### Pass 4: Dependencies Correctness & Final Unused Code Sweep
Run a final deep scan for:

- **Dependency correctness**:
  - Check for version conflicts (e.g., two different versions of `kotlin-stdlib`).
  - Remove any `implementation` dependencies that are never used (transitive or explicit).
  - Ensure `testImplementation` and `androidTestImplementation` dependencies are confined to test source sets.
- **Type‑safety & import accuracy** (critical to prevent compilation errors):
  - **Override return types**: Every `override` function must have a return type identical to the supertype’s declaration. If the super method returns `Unit`, your override must return `Unit` (or omit the return type). Fix any mismatches immediately.
  - **Floating‑point consistency**: Kotlin does **not** coerce `Float` and `Double`. Verify that numeric literals use explicit suffixes (`f` for `Float`, `d` or no suffix for `Double`) and that explicit `.toFloat()`/`.toDouble()` conversions are used when passing values to functions expecting a different type. Ensure all arithmetic results match the expected type.
  - **Explicit type declarations**: When the right‑hand side is ambiguous (e.g., numeric literals, generic calls), declare the variable type explicitly to avoid inference mismatches.
  - **Import verification**: For every external reference, confirm the correct import is present. Specifically:
    - `clickable` → `import androidx.compose.foundation.clickable`
    - `ContentScale` → `import androidx.compose.ui.layout.ContentScale`
    - `setViewTreeLifecycleOwner` → `import androidx.lifecycle.setViewTreeLifecycleOwner`
    - `SharedPreferences.Editor` → use fully‑qualified `android.content.SharedPreferences.Editor` (never `Preferences.Editor`; if using DataStore, import `androidx.datastore.preferences.core.Preferences.Editor` instead).
    - For any other unresolved reference, add the correct import or, if the dependency is missing, add a comment indicating the required dependency.
- **Unused‑code sweep** (aggressive):
  - Scan all `val`/`var` declarations – delete any that are assigned but never read.
  - Scan all functions – delete any private functions that have no callers.
  - Scan all classes – flag any `data class` or regular class that is never instantiated (except for DTOs used purely in JSON serialization, which must be explicitly whitelisted).
  - Check for unresolved references / import typos that would cause compilation errors (e.g., `import com.exmaple` → fix to `com.example`).

## 🧠 Core Operating Principles (Enforced Across All 4 Passes)

When analyzing or modifying any file, strictly enforce these rules:

1. **Static Analysis & Clean Code**
   - Complexity: If a function exceeds 15 lines or cyclomatic complexity > 10, break it down.
   - Naming: Classes = nouns (`UserRepository`), functions = verbs (`fetchUser`). Booleans start with `is`/`has`/`should`.
   - Null Safety: Never use `!!`. Use `?.`, `?:`, or `requireNotNull()` with a message.
   - Unused Code: Delete immediately (reinforced in Pass 4).
   - Magic Numbers: Extract to `const val` at top of file or companion object.

2. **Code Formatting (ktlint)**
   - Max line length = 120 characters. Use trailing commas for multiline params.
   - 4‑space indentation. No tabs.
   - Space after keywords (`if (`, `for (`).
   - Modifier order: `public/private/internal` → `suspend` → `override` → `fun`.

3. **Dependency Management (Version Catalog)**
   - Never hardcode versions in module `build.gradle.kts`. Use `libs.versions.toml`.
   - If a dependency is missing from the TOML, add it before using it.

4. **Architecture & Layering**
   - `domain` MUST NOT import `android.*`, `Compose`, or `ViewModel`.
   - `data` depends only on `domain`.
   - `presentation` may depend on `domain` but MUST NOT directly import `data` (use interfaces).
   - Break cyclic dependencies by moving interfaces into `domain`.

5. **Error Handling & Robustness**
   - Network/database operations must return `sealed class Result<out T> { Success, Error }`. No raw exceptions for expected failures.
   - Coroutines: use `viewModelScope` / `lifecycleScope`; unit tests must use `runTest`.
   - Immutability: prefer `val` and `data class.copy()`.

6. **Testing**
   - New business logic MUST come with a unit test (JUnit + MockK).
   - Bug fixes MUST include a regression test.
